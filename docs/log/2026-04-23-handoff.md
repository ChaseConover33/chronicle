## Decisions Made This Session

All major design decisions are captured in `docs/decisions.md` as ADRs. The 8 decisions recorded are:

1. **One dataset, many lenses** — universal capture format, personalization happens in analysis layer
2. **Markdown for storage** — not HTML, not rich text JSON
3. **SQLite, not Postgres** — single-file, trivial backup, fits single-user workload
4. **Next.js (not FastAPI + separate frontend)** — one deployment unit, TypeScript end-to-end, SSR for public pages
5. **AI preserves voice, does not rewrite** — organize and fix typos only, never elevate vocabulary or change sentences
6. **Summaries are entries** — same table, distinguished by `type` field, enables recursive summarization
7. **Draft/published workflow with soft lock** — edits to published entries require confirmation
8. **Domains and lenses are independent axes** — domains = *what part of life*, lenses = *how to interpret*. Orthogonal systems.

The lens/domain distinction emerged mid-conversation. The user's insight was that people aren't one thing — a Christian software engineer in therapy shouldn't have to pick one template. This reshaped the whole architecture.

## What Worked

- **Documentation-first approach.** Writing vision → architecture → decisions → full project spec before any code forced clarity on the design. The user explicitly valued this over "start coding and see what happens."
- **Structured ADRs with alternatives considered.** The "what / why / alternatives / consequences" format from the self-hosted project was applied here and produced high-signal decision records.
- **Showing tradeoffs and asking before committing to them.** The user made several design shifts during the conversation (added domains as a separate axis from lenses, added retrospective journaling, added custom templates). The iterative back-and-forth surfaced better designs than a first pass would have.
- **Stack change from FastAPI to Next.js.** Originally proposed FastAPI + SQLite + HTMX. User pushed back on HTMX for the frontend. Shifted to Next.js + shadcn/ui + Drizzle ORM, and this is the right choice — single deployment unit, SSR for future public display on chaseconover.com.
